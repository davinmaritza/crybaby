# CryBaby — Custom RMM Platform Spec

> Build spec for an AI coding agent (Claude Opus in Antigravity). Implement in phases as ordered below. Ask no clarifying questions that are already answered here — pick sensible defaults elsewhere and note assumptions in code comments.

## 1. Project Summary

CryBaby is a custom, self-hosted RMM (Remote Monitoring & Management) platform for managing a fleet of Windows 10/11 servers (currently 40, growing). It replaces off-the-shelf tools (Tactical RMM, MeshCentral) with a purpose-built system that also exposes an AI agent layer for natural-language fleet operations.

**Owner context:** Fleet is used for VPS/game server hosting (ByteNodes). Command center backend will be deployed as an app on an existing Pterodactyl panel (via Wings), reachable over a normal exposed port like any other Pterodactyl-hosted server.

## 2. High-Level Architecture

```
[Windows 10/11 Agent] <--WebSocket (TLS)--> [Command Center Backend] --- SQLite/Postgres
        (x40+)                                      |
                                    +-----------------+-----------------+
                                    |                 |                 |
                              [Web Dashboard]   [Telegram Bot]   [AI Agent Layer]
```

- **Agent**: native binary installed on each Windows server. Persistent outbound WebSocket connection to backend (works through NAT without inbound ports). Sends heartbeat + system spec on connect, executes commands pushed from backend.
- **Command Center Backend**: single service, deployed via Pterodactyl. Owns the WebSocket server, REST API, database, auth, and the AI tool-calling layer.
- **Web Dashboard**: consumes the same REST/WebSocket API. Fleet summary + per-server table + fleet core map (already prototyped, see `crybaby-dashboard.html` reference for visual direction — dark ops-room theme, Chakra Petch + IBM Plex Mono, amber/teal/red status colors).
- **Telegram Bot**: consumes the same internal API as the dashboard. Whitelisted user IDs only.
- **AI Agent Layer**: backend module that takes natural-language input (from Telegram or a chat panel in the dashboard) and calls internal tools (`exec_command`, `get_status`, `transfer_file`, `list_servers`, etc.) via Claude API tool use, then replies in natural language.

## 3. Tech Stack (defaults — deviate only with good reason)

- Agent: **Go** (single static binary, cross-compiles for Windows, small footprint)
- Backend: **Go** (or Node.js if faster to iterate — pick one and stay consistent), WebSocket via `gorilla/websocket` or equivalent
- Database: **SQLite** for now (single file, fine at this scale), structured so migration to Postgres later is easy
- Web dashboard: plain HTML/CSS/JS or React, dark theme per the existing prototype
- Telegram bot: `telegraf` (Node) or `go-telegram-bot-api` (Go), matching backend language
- Transport: WebSocket over TLS for agent<->backend; REST + WebSocket for dashboard<->backend

## 4. Core Data Model

```
Server
  id            (uuid, primary key — generated by backend on first registration)
  hostname      (reported by agent, not trusted as unique identity)
  custom_name   (nullable — user-assigned label, e.g. "Game Node BDG-01")
  cluster_id    (nullable, FK -> Cluster)
  os_version    ("Windows 10" / "Windows 11" + build)
  cpu_model
  cpu_cores
  cpu_threads
  ram_total_mb
  disk_total_mb
  agent_version
  status        (online / offline / pending_approval)
  first_seen_at
  last_seen_at
  approved_at   (nullable — see 5.1 new-device flow)

Cluster
  id
  name          (user-defined, e.g. "Bandung DC", "Game Hosting Pool")
  description

MetricSample (time-series, keep recent window only e.g. 7 days, prune older)
  server_id
  timestamp
  cpu_load_pct
  ram_used_mb
  disk_used_mb
  uptime_seconds

CommandLog
  id
  server_id
  issued_by       (dashboard user / telegram user id / ai_agent)
  command
  result
  issued_at
  completed_at
```

## 5. Feature Spec

### 5.1 New device detection + approval flow
- When an agent connects with an `id` the backend has never seen, backend creates a `Server` row with `status = pending_approval` and does **not** treat it as part of the active fleet yet.
- Backend pushes a notification immediately to:
  - Telegram (all whitelisted admins): `🆕 New device detected: <hostname> (<cpu_cores>c/<cpu_threads>t, <ram_total>GB RAM, <os_version>) — approve with /approve <id> or reject with /reject <id>`
  - Web dashboard: a "Pending Devices" panel, separate from the main fleet table.
- Only after explicit approval (dashboard click or `/approve <id>`) does the server show up in the main fleet view, summary totals, and core map.
- Rejecting a device tells the agent to uninstall itself (or at minimum stop retrying) — don't leave rejected agents endlessly reconnecting.

### 5.2 Delete / decommission a server
- Action available from dashboard (per-row action) and Telegram (`/remove <id>`).
- On delete: mark `Server` row as removed (soft delete — keep historical `CommandLog`/`MetricSample` for audit unless user explicitly asks for hard delete), close the WebSocket connection, and instruct the agent to stop retrying reconnection (agent should not resurrect itself as a "new device" unless manually reinstalled).
- Confirm before destructive delete in both dashboard and bot (bot: require `/remove <id> confirm`).

### 5.3 View server specs
- Per-server detail view (dashboard: click a row; bot: `/info <id>`) shows: hostname, custom name, cluster, OS + build, CPU model/cores/threads, RAM total, disk total, agent version, uptime, first seen, last seen.
- This is static spec data reported once at registration/reconnect, distinct from the live `MetricSample` time-series (load/usage).

### 5.4 Naming servers
- `custom_name` field, editable any time from dashboard (inline edit) or bot (`/rename <id> <new name>`).
- Display priority everywhere in UI: `custom_name` if set, else `hostname`.

### 5.5 Clustering
- Servers can be grouped into a `Cluster` (e.g. by physical location, purpose, or client).
- Dashboard: cluster is a filter/grouping dimension — summary strip should support "All fleet" totals as well as per-cluster totals (toggle or dropdown).
- Cluster CRUD: create/rename/delete cluster, assign/reassign servers to a cluster (dashboard drag-or-select; bot: `/cluster set <server_id> <cluster_name>`).
- A server with no cluster shows under an "Unassigned" group.

### 5.6 Dashboard (reference build exists — extend, don't redesign)
- Keep the dark ops-room visual direction already built (Chakra Petch display font + IBM Plex Mono data font, graphite background, amber/teal/red status colors, Fleet Core Map as the signature element).
- Add: Pending Devices panel (5.1), per-server detail drawer (5.3), inline rename (5.4), cluster filter + per-cluster summary (5.5), delete action with confirm (5.2).
- Fleet Core Map should respect cluster filtering — when a cluster is selected, only show threads belonging to that cluster's servers.

### 5.7 Telegram bot commands
```
/status                 - fleet summary (cores, threads, RAM, disk, online/offline)
/status <cluster>       - summary scoped to one cluster
/list                   - all servers with short status line
/list <cluster>         - servers in one cluster
/info <id>              - full spec + live stats for one server
/pending                - list devices awaiting approval
/approve <id>           - approve a pending device
/reject <id>            - reject/uninstall a pending device
/rename <id> <name>     - set custom name
/remove <id>            - soft-delete a server (requires "confirm" arg)
/cluster list           - all clusters with server counts
/cluster new <name>     - create cluster
/cluster set <id> <name>- assign server to cluster
/exec <id> <command>    - run a shell command, return output
/broadcast <command>    - run on all online servers
/file get <id> <path>   - fetch a file from a server
/file put <id> <path>   - push a file to a server (attach file in Telegram message)
```
- Auth: hardcoded/config whitelist of Telegram numeric user IDs. Reject and log any command from a non-whitelisted user; do not reveal fleet info to them.

### 5.8 AI agent layer
- Natural-language entry point (Telegram free-text not matching a `/command`, and a chat box in the dashboard).
- Backend calls Claude API (`/v1/messages`) with tool definitions matching the internal API: `get_fleet_status`, `get_server_info`, `list_servers(cluster?)`, `exec_command(server_id, command)`, `transfer_file(...)`, `rename_server(...)`, `set_cluster(...)`.
- Model decides which tool(s) to call based on the request (e.g. "server mana yang RAM-nya udah di atas 90%?" → calls `get_fleet_status`/`list_servers`, filters, replies in natural language).
- Every AI-initiated `exec_command` gets logged in `CommandLog` with `issued_by = ai_agent` plus the original natural-language prompt, for auditability.
- Guardrail: AI agent layer must not auto-approve pending devices or auto-delete servers without an explicit user instruction naming the target — no autonomous fleet changes on its own initiative.

## 6. Agent Behavior (Windows 10/11 binary)

- On first run: generate/persist a local UUID (registry or local file), collect system spec (CPU model/cores/threads via WMI, RAM total, disk total, OS version/build), connect to backend WebSocket, send registration payload.
- Heartbeat interval: every 5–10s while idle (not more frequent — keep resource use low), event-driven push for command results.
- Resource budget target: idle RAM ~20–50MB, ~0% idle CPU.
- Auto-update: on connect, compare local `agent_version` against version backend advertises; if newer available, pull and self-replace.
- Modules (plugin-style, core stays minimal):
  - `sysinfo` — spec + live metrics
  - `shell` — exec command, stream stdout/stderr back
  - `filetransfer` — chunked upload/download (don't load full file into RAM)
- On backend-issued "remove" instruction: stop reconnect loop, optionally self-uninstall.

## 7. Install / Deployment

- Primary install path: `irm https://cdn.bytenodes.icu/crybaby-install.ps1 | iex` (hosted installer script, pulls latest agent binary + registers).
- Secondary: flashdisk-based manual install for air-gapped/first-batch machines.
- Mass-deploy option for servers with existing WinRM/admin access: PowerShell remoting loop or Ansible playbook to push + run installer across all hosts at once.
- Backend: deployed as a Pterodactyl egg/app on existing Wings infrastructure — needs one exposed port for the WebSocket/REST listener, persistent volume for the SQLite file (or Postgres connection if that's chosen instead).

## 8. Security Requirements

- All agent<->backend and dashboard<->backend traffic over TLS.
- Agent auth: per-device token issued at approval time, sent on every reconnect (not just device UUID, which is not itself a secret).
- Dashboard auth: login-gated, not open on the internet without credentials.
- Telegram bot: numeric user ID whitelist, config-driven, reject-and-log anything else.
- Command execution audit trail: every `exec_command`/`file` action logged with who issued it (dashboard user, Telegram user id, or `ai_agent` + prompt).

## 9. Build Order (suggested phases)

1. Backend core: DB schema, WebSocket server, agent registration + heartbeat, REST API skeleton
2. Agent: sysinfo + heartbeat + shell exec module, Windows binary, install script
3. Pending-device flow + approve/reject (5.1)
4. Web dashboard: fleet summary, server table, fleet core map, wired to real API (replace the mock data in the existing prototype)
5. Naming, clustering, delete (5.2, 5.4, 5.5)
6. File transfer module (agent + backend + dashboard UI)
7. Telegram bot: full command set (5.7)
8. AI agent layer wired to Claude API with tool definitions (5.8)
